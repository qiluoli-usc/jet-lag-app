# Jet Lag App — 阶段二任务与行动方案（2026-03-16）

## 文档目的

本文档用于把“阶段二：核心功能补全”从早期 Gap 分析中的粗粒度条目，整理成当前仓库可直接执行的任务清单和行动方案。

整理依据：

- `docs/GAP_ANALYSIS_2026-03-12.md`
- `docs/JetLag_HideSeek_APP_SPEC_CN_v3.md`
- `docs/JetLag_HideSeek_APP_PRD_CN_v3_Codex.extracted.txt`
- 当前仓库实现状态（截至 2026-03-16）

## 阶段二目标

阶段二的目标，不再是基础设施，而是把“游戏主流程中仍依赖外部工具或缺少 UI 闭环”的核心模块补齐，使产品从“能玩原型”推进到“核心内置能力基本完整”。

按原始文档，阶段二主要包括：

1. 地图全功能集成
2. 相机 + 证据上传 + 存储
3. In-App 聊天
4. Summary 复盘完善

但结合当前仓库状态，阶段二需要按真实情况重新排序。

## 当前状态判断

### 已经不是阶段二主阻塞的部分

- `Summary` 已不再是早期那种空骨架。
  - 当前移动端已经有：
  - 时间摘要
  - seeker 路线地图
  - 时间线
  - 下一轮重置入口

因此，`Summary` 在阶段二中应该从“从零建设”调整为“结构和内容补强”。

### 仍然是阶段二主阻塞的部分

- 地图模块仍只有可用骨架，缺少 Lobby 配置和完整推理工具链。
- 证据上传只有后端骨架，没有移动端拍摄 / 上传 / 查看闭环。
- In-App 自由聊天仍然完全缺失。

### 需要提前拉入阶段二的依赖项

虽然原始 Gap 分析把“争议仲裁投票 UI”放得更后，但按当前代码判断，下面两条能力如果没有最小仲裁 UI，会导致阶段二本身难以闭环：

- POI 合法性争议（`review_count` override）
- 证据确认 / 驳回 / 仲裁

因此，建议把“最小仲裁投票 UI”提前并入阶段二，而不是等到阶段三。

## 阶段二任务清单

### 任务 2.1：地图模块补全

目标：

- 把当前 `Seeking` 阶段中“可搜索 POI + 可画 polygon”的能力，补成“房间级统一地图工具”。

当前已具备：

- 后端已有 `mapProvider` / `transitPackId` 字段
- 后端已有 `places/search`、`places/details`、`admin-levels/reverse`
- 前端已有 POI 搜索、位置上报、共享 polygon 标注

仍需完成：

- Lobby 中增加房间级地图配置 UI
  - `mapProvider` 选择
  - `transitPack` 选择
  - 边界 / 躲藏区绘制入口
- Seek 阶段补齐地图工具
  - 距离测量
  - 圆形 / 线段 / 多边形标注
  - 标注图层管理
  - POI 详情查看
  - 行政区层级查询面板
- Web 与 mobile 保持同一套基础能力
- 房间日志里明确记录 mapProvider / transitPack 生效情况

验收标准：

- 房主创建房间时可明确选择地图来源与 transit pack
- seeker 可在房间内完成测距、圈地、查 POI、看 place details，不依赖外部地图 App
- 同一房间中不同端看到的 mapProvider 和共享标注一致

### 任务 2.2：相机 + 证据上传 + 存储闭环

目标：

- 把当前只有后端接口骨架的 evidence 流程，补成“可拍摄、可上传、可查看、可用于裁决”的完整链路。

当前已具备：

- 后端已有 `evidence/upload-init`
- 后端已有 `evidence/complete`
- 房间状态中已有 `evidence` 结构与事件

仍需完成：

- 设计统一证据存储适配层
  - 本地开发可先走文件系统或 mock storage
  - 接口设计保持 S3 / 对象存储兼容
- 移动端增加证据 UI
  - 拍摄 / 选择图片
  - 预览
  - 上传进度
  - 上传失败重试
- 将 photo 类问题、诅咒证据、抓捕争议接到同一证据流
- 增加证据查看页 / 列表 / 元数据展示
  - 时间
  - 上传者
  - 关联 question / curse / dispute

验收标准：

- hider 可在移动端完成一次拍摄并上传证据
- 房间内能看到证据记录及其关联事件
- 证据不再只是内存对象，而有明确的存储 key 与查看入口

### 任务 2.3：In-App 聊天与线索流

目标：

- 在结构化问答之外，补齐房间内的自由聊天和线索流，避免继续依赖外部 IM。

当前已具备：

- 结构化提问 / 回答主链路已完成
- 后端已有 `clues` 能力骨架

仍需完成：

- 设计聊天消息数据结构
  - 普通聊天消息
  - 系统消息
  - clue 消息
- 增加 REST / WebSocket 消息通道
- 移动端与 Web 端增加聊天面板
- 将 hider clue 统一收口到同一消息流中，而不是单独悬空存在
- 增加未读提示、时间戳、发送失败提示

验收标准：

- 同房间玩家可在 App 内发送和接收实时文本消息
- 聊天与结构化问答区分明确，不混淆“官方提问”
- clue 在房间流中可见且可回放

### 任务 2.4：Summary 复盘补强

目标：

- 在现有 Summary 基础上，把“路线回放 + 时间摘要”扩展成真正可复盘的一局记录。

当前已具备：

- 基本胜负摘要
- seeker 路线图
- 路线统计
- 时间线
- 下一轮重置

仍需完成：

- Summary 中按模块展示
  - 问答记录
  - 卡牌使用记录
  - 诅咒历史
  - 抓捕与争议
  - 证据列表
- 时间线内容更可读
  - 按事件类型分组
  - 关键事件高亮
- 与后端 summary payload 对齐
  - 减少前端靠原始 event JSON 生拼的部分

验收标准：

- 一局结束后，不看原始 log 也能理解问答、卡牌、抓捕和结果
- Summary 成为“复盘视图”，而不是“事件 JSON 浏览器”

### 任务 2.5：最小仲裁投票 UI（建议提前并入阶段二）

目标：

- 为地图合法性与证据争议提供最小可用仲裁界面，避免阶段二核心能力只能停在半成品。

当前已具备：

- 后端已有 `disputes` 创建、投票、解决接口
- 事件流中已有 `dispute.created / voted / resolved`

仍需完成：

- 移动端与 Web 端的争议列表 / 投票 UI
- 在 POI 合法性与证据流中增加“发起争议”入口
- 展示投票结果与最终决议

验收标准：

- 玩家能在客户端看到当前 dispute
- 具备资格的玩家能投票并看到结果
- POI override 或证据结论能在房间内被锁定

## 不建议塞进阶段二的内容

这些内容在原始文档中存在，但我不建议和本阶段并行推进：

- 完整 30+ 张诅咒逻辑
- 完整 GTFS 导入器
- Wikipedia / In-App Reference 全量面板
- Referee 独立 Web 端
- 多局统计仪表盘

原因很简单：

- 它们会显著放大范围
- 对“核心功能补全”不是最短闭环
- 会稀释当前最缺的地图 / 证据 / 聊天三条主线

## 阶段二行动方案

### 原则

阶段二不建议按“模块负责人各自并行”推进，而应该按“用户闭环”推进。

建议顺序：

1. 先补地图房间配置与 Seek 工具
2. 再补证据流与最小仲裁
3. 再补聊天
4. 最后补 Summary 内容整合

这样做的原因：

- 地图是 Seek 阶段的主工作区，必须先稳定
- 证据与仲裁依赖地图 / 题目 / 抓捕场景
- 聊天接在事件流之后实现最顺
- Summary 适合最后统一消化前面新增的数据结构

### 建议执行顺序

#### 步骤 A：地图与房间配置（优先级最高）

先做：

- 梳理房间配置对象与客户端表单
- 在 Lobby 中接入 `mapProvider` / `transitPack`
- 补地图工具栏和共享图层
- 补 mobile / web 两端的基本一致性

交付物：

- Lobby 可配置地图来源
- Seeking 可完成测距 / 标注 / POI / place detail 基础闭环

#### 步骤 B：证据流与最小仲裁

再做：

- 统一 evidence 数据模型
- 接入移动端拍摄 / 上传
- 增加证据查看面板
- 增加最小 dispute 投票 UI

交付物：

- photo / curse / catch 相关流程可真正附带证据
- 房间内能发起并完成最小仲裁

#### 步骤 C：聊天与 clue 收口

再做：

- 消息模型
- 服务端消息广播
- 前端聊天面板
- clue 融合到消息流

交付物：

- 房间内不再需要外部聊天工具

#### 步骤 D：Summary V2

最后做：

- 统一 summary payload
- 将问答 / 卡牌 / 诅咒 / 证据 / 争议汇总到 Summary
- 调整移动端和 Web 端复盘可读性

交付物：

- Summary 成为阶段二闭环后的统一复盘页

## 建议的阶段二里程碑

### 里程碑 2A：地图闭环

- Lobby 地图配置完成
- Seeking 地图工具完成第一版
- Web / mobile 同步可用

### 里程碑 2B：证据闭环

- 拍摄 / 上传 / 查看跑通
- 最小仲裁投票跑通

### 里程碑 2C：沟通与复盘闭环

- 聊天跑通
- Summary V2 跑通

## 当前建议结论

如果按当前仓库状态来判断，阶段二最合理的主线不是“把 Gap 分析原文逐条照搬”，而是：

1. 地图模块补全
2. 证据上传闭环
3. 聊天闭环
4. Summary 补强
5. 把最小仲裁 UI 提前拉入本阶段

这是当前最短、也最像产品闭环的推进顺序。
